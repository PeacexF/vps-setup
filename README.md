# что после запуска
Ваша текущая сессия
Когда вы запускаете скрипт, вы уже находитесь в системе под пользователем root (или через sudo). Запуск скрипта не разрывает текущее соединение. Даже после того, как скрипт изменит порт SSH, отключит вход по паролю и создаст нового пользователя, ваша открытая консоль останется активной и сохранит права root.

Это ваше «золотое окно»: пока вы не закрыли этот терминал, вы можете исправлять любые ошибки, менять конфиги обратно или пересоздавать пароли.

2. Новый пользователь и sudo
Скрипт выполняет команду usermod -aG sudo $NEW_USER. Это означает, что созданный пользователь secureuser:

Имеет право выполнять команды от имени суперпользователя.

Для этого вам нужно будет вводить sudo команда (система может запросить пароль пользователя, если он был установлен, или пустить так, если настроен беспарольный доступ, но в данном скрипте пароль лучше задать вручную командой passwd secureuser перед выходом).

3. Главная опасность
Скрипт меняет порт SSH на 2222 и запрещает вход по паролю. Если вы закроете текущую сессию root, а новая сессия под secureuser не заработает (например, из-за того, что ключ не подошел или порт закрыт провайдером VPS), вы окажетесь заблокированы.

Что нужно сделать СРАЗУ после завершения скрипта (НЕ закрывая текущее окно):

Задайте пароль новому пользователю (на случай, если sudo его потребует):

Bash
passwd secureuser
Проверьте статус Firewall:

Bash
ufw status
Убедитесь, что там есть строка 2222/tcp ALLOW.

Проверьте SSH-подключение в НОВОМ окне терминала, не закрывая старое:

Bash
ssh -p 2222 secureuser@ip_вашего_сервера



# как копировать
Для работы с приватными репозиториями на сервере есть два основных профессиональных способа: SSH-ключи (Deploy Keys) и Personal Access Tokens (PAT).

Способ 1: SSH-ключи (Deploy Keys) — Рекомендуемый
Это самый безопасный способ. Вы создаете отдельный ключ на сервере и даете ему доступ только к одному конкретному репозиторию на GitHub/GitLab.

На сервере (под вашим новым пользователем) создайте ключ:

Bash
ssh-keygen -t ed25519 -C "vps-deploy-key"
(На все вопросы жмите Enter).

Выведите публичный ключ на экран:

Bash
cat ~/.ssh/id_ed25519.pub
В интерфейсе GitHub/GitLab:

Зайдите в свой приватный репозиторий.

Перейдите в Settings -> Deploy keys.

Нажмите Add deploy key.

Вставьте скопированный текст ключа и дайте ему название (например, "My VPS").

Клонируйте репозиторий через SSH-адрес:

Bash
git clone git@github.com:ваш_логин/ваш_репозиторий.git
Способ 2: Personal Access Token (PAT)
Если вы не хотите возиться с ключами, можно использовать токен вместо пароля при клонировании через HTTPS.

На GitHub: Settings -> Developer settings -> Personal access tokens -> Tokens (classic).

Сгенерируйте новый токен с правами repo. Обязательно сохраните его, он покажется один раз.

На сервере клонируйте так:

Bash
git clone https://ВАШ_ТОКЕН@github.com/username/repo.git
Минус: токен может сохраниться в истории команд (history), что менее безопасно.

Как это интегрировать в ваш процесс
Поскольку вы только что настроили Docker, скорее всего, вы захотите развернуть там проект. Лучшая практика выглядит так:

Вы запускаете свой bash-скрипт (он создает пользователя и ставит Docker/Git).

Заходите под новым пользователем secureuser.

Генерируете Deploy Key (как в Способе 1).

Добавляете его в GitHub.

Делаете git clone.

Запускаете проект через docker compose up -d.

Что еще можно сделать для безопасности
Раз уж мы заговорили про приватность и Docker:

Docker Content Trust: Можно включить проверку подписей образов, чтобы Docker скачивал только проверенные слои (export DOCKER_CONTENT_TRUST=1).

Secrets: Никогда не храните пароли и API-ключи прямо в коде репозитория. Используйте файл .env, который добавлен в .gitignore. На сервере создавайте этот файл вручную или через защищенные хранилища.

Fail2Ban для кастомных сервисов: Если вы напишете свой API на Python/FastAPI, можно настроить Fail2Ban так, чтобы он банил за слишком частые ошибки 404 или попытки подбора пароля к вашему приложению, а не только к SSH.
